# 将来拡張

claude-code-usage-barのMVP（v1.0）以降に検討する拡張機能。[MVPスコープ](mvp-scope.md)のv1.1/v2.0候補を深掘りし、各項目の目的・実装アプローチ・考慮事項を整理する。

---

## OpenTelemetry連携

### OTEL_LOG_RAW_API_BODIESとの統合

**目的**: `statusLine`では取得できないAPI呼び出しの詳細情報（トークン内訳、モデル別コスト、レイテンシ）を補完し、組織レベルでの使用量監視を実現する。個人向けメニューバー表示の主データソースは引き続き`statusLine`とし、OpenTelemetryは組織監視・分析向けの補助データソースとして位置づける。

**実装アプローチ**:

Claude Codeは以下のOTel機能を提供しており、これらを活用する:

| Claude Code OTel機能 | 環境変数 | 取得データ |
|---------------------|---------|-----------|
| メトリクスエクスポート | `OTEL_EXPORTER_OTLP_ENDPOINT` | 使用量メトリクス |
| ログエクスポート | `OTEL_EXPORTER_OTLP_ENDPOINT` | セッションログ |
| トレースエクスポート | `OTEL_EXPORTER_OTLP_ENDPOINT` | API呼び出しスパン |
| Raw APIボディ | `OTEL_LOG_RAW_API_BODIES=true` | リクエスト/レスポンス全文 |

### メトリクス送信

collectorが収集した使用率データをOTELメトリクスフォーマットで外部にエクスポートする。

```
claude_code.rate_limit.five_hour.used_percentage  gauge  42.0
claude_code.rate_limit.seven_day.used_percentage   gauge  18.5
claude_code.context_window.used_percentage         gauge  22.5
claude_code.session.cost_usd                       counter 0.42
```

| エクスポート先 | 方式 | 備考 |
|--------------|------|------|
| Grafana | OTLP/gRPC or OTLP/HTTP | ダッシュボードでの可視化 |
| Prometheus | /metrics エンドポイント | pull型モニタリング |
| ローカルファイル | JSONL | オフライン分析用 |

**考慮事項**:

- OTEL連携はネットワーク通信を伴うため、MVP（v1.0）の「ネットワーク通信なし」原則からの拡張となる
- ユーザーが明示的にオプトインした場合のみ有効化し、デフォルトは無効
- `statusLine`が主データソース、OTelは補助という階層を崩さない。OTelバックエンドが停止してもメニューバー表示に影響しない設計とする
- 組織で複数メンバーのClaude Code使用量を集約する用途を想定

## 履歴分析

### ccusage互換JSONL解析

**目的**: Claude Codeが`~/.claude/projects/`配下に蓄積するローカルJSONLを解析し、コスト・使用量の長期トレンドを把握する。[ccusage](https://github.com/ryoppippi/ccusage)との互換性を持つことで、既存の分析ワークフローとの統合も実現する。

**実装アプローチ**:

| 項目 | 詳細 |
|------|------|
| 入力形式 | JSONL（1行1レコード） |
| 対象ディレクトリ | `~/.claude/projects/`配下のセッションログ |
| 解析項目 | トークン使用量、コスト、モデル、セッション時間 |

**考慮事項**:

- Claude CodeのローカルJSONLフォーマットはバージョン間で変更される可能性がある。パーサーにバージョン検出と後方互換ロジックを含める
- 大量のJSONL解析はメモリ消費に注意。ストリーミング読み込みを採用する

### 日次・週次レポート

**目的**: 蓄積した履歴データから定期レポートを生成し、使用パターンの自己認識を支援する。

**実装アプローチ**:

| レポート種別 | 内容 | 表示場所 |
|------------|------|---------|
| 日次サマリ | 当日のトークン使用量・推定コスト・ピーク使用率 | ポップオーバー |
| 週次サマリ | 7日間の使用パターン・曜日別トレンド | ポップオーバー（拡張ビュー） |
| CSV/JSONエクスポート | 任意期間の生データ出力 | メニュー操作 |

### 5時間ブロック単位の使用パターン可視化

**目的**: 5時間枠のローリングウィンドウに合わせた使用率推移を可視化し、「いつ集中的に使ったか」「どの時間帯にピークが来るか」をヒートマップ/グラフで把握する。

**実装アプローチ**:

Swift Chartsでグラフ表示する。ヒートマップ形式も検討。

```swift
import Charts

Chart(usageHistory) { entry in
    LineMark(
        x: .value("Time", entry.timestamp),
        y: .value("Usage", entry.fiveHourPercentage)
    )
    .foregroundStyle(colorForPercentage(entry.fiveHourPercentage))
}
```

| 表示項目 | 詳細 |
|---------|------|
| X軸 | 時刻（直近24時間〜7日間） |
| Y軸 | 使用率（0-100%） |
| 色分け | 通常（緑）/ 注意（オレンジ）/ 警告（赤）の閾値ライン |
| ヒートマップ | 曜日×時間帯のマトリクス、色濃度で使用率を表現 |
| インタラクション | ホバーで時刻・使用率の詳細表示 |

**考慮事項**:

- 履歴データの蓄積にはSwiftDataまたはappend-only JSONLを使用（[tech-decisions.md](tech-decisions.md)で再評価技術として記載）
- ポップオーバーのサイズ制約内でグラフを見やすく配置する必要がある

## 複数プロファイル

### Claude Code設定プロファイルの切り替え

**目的**: 複数のワークスペースやプロジェクトを使い分けるユーザー向けに、プロファイル単位でコスト・使用率を追跡する。

**実装アプローチ**:

Claude Codeは`~/.claude/settings.json`（グローバル）と`.claude/settings.json`（プロジェクト）の2階層で設定を管理する。

| 項目 | 詳細 |
|------|------|
| プロファイル識別 | ワークスペースパスまたはユーザー定義のラベル |
| 切り替え方式 | ポップオーバーのドロップダウン |
| データ分離 | プロファイルごとに履歴ファイルを分離 |

### プロファイル別使用率追跡

- 各プロファイルの`session_id`をグループ化
- プロファイル別のコスト集計
- アクティブプロファイルをメニューバーに表示（オプション）

**考慮事項**:

- `rate_limits`はアカウント単位で共通のため、プロファイル間で共有される。プロファイル別に分離できるのは`context_window`と`cost`のみ
- プロファイル数の上限を設定し、UI/メモリ消費を管理する

## ウィジェット対応

### macOS Widget Extension

**目的**: メニューバーをクリックせずとも、デスクトップやNotification Centerで使用率をグランス（ちら見）確認できるようにする。

**実装アプローチ**:

macOS 14+のWidget Extensionを使用する。

| サイズ | 表示内容 |
|--------|---------|
| small | 5時間枠の使用率ゲージ + リセットカウントダウン |
| medium | 5時間枠 + 7日枠のゲージ + リセットカウントダウン + モデル名 |

```swift
struct UsageWidget: Widget {
    var body: some WidgetConfiguration {
        StaticConfiguration(kind: "usage", provider: UsageTimelineProvider()) { entry in
            UsageWidgetView(entry: entry)
        }
        .configurationDisplayName("Claude Code Usage")
        .description("Monitor your Claude Code rate limits")
        .supportedFamilies([.systemSmall, .systemMedium])
    }
}
```

### ロック画面ウィジェット

将来的にmacOSでロック画面ウィジェットがサポートされた場合に対応を検討。iOS連携（iPhoneのロック画面ウィジェットからMacのClaude Code使用率を確認）の余地もある。

| 項目 | 詳細 |
|------|------|
| 表示形式 | circular / rectangular |
| 更新頻度 | WidgetKitのタイムラインに依存（最短15分） |

**考慮事項**:

- WidgetKitの更新間隔（最短15分）は`statusLine`の`refreshInterval`（10秒）より大幅に長い。ウィジェットは「ざっくり確認」用途として位置づけ、リアルタイム情報はメニューバーのポップオーバーに誘導する
- Widget Extensionはメインアプリとは別プロセスで動作するため、App Groupsを使用してデータを共有する必要がある
- iOS連携を視野に入れる場合、iCloud経由でのデータ同期が必要となり、ネットワーク通信原則との整合を慎重に検討する

## その他の拡張候補

### Claude service statusの統合

**目的**: Anthropicのサービス障害時にユーザーへ即座に状況を伝え、「自分の設定の問題か、サービス側の問題か」の切り分けを支援する。

**実装アプローチ**:

Anthropicのステータスページ（status.anthropic.com）の情報をポップオーバーに統合する。

| 状態 | 表示 |
|------|------|
| 正常稼働 | 表示なし（通常時はノイズを避ける） |
| 一部障害 | オレンジのインジケーター + 障害内容サマリ |
| 大規模障害 | 赤のインジケーター + 「Claude Codeが利用できない可能性があります」 |

**考慮事項**: ネットワーク通信を伴うため、OTEL連携と同様にオプトイン方式とする。

### ピーク時間予測

**目的**: 現在の使用ペースから制限到達時刻を予測し、ペース配分の判断材料を提供する。

**実装アプローチ**:

履歴データから使用率の増加率を算出し、線形外挿で予測する。

```
現在のペース: 8.4%/時
予測制限到達: 約6.9時間後（本日 19:42頃）
推奨: ペースを6%/時以下に抑えると5時間枠内に収まります
```

| 項目 | 詳細 |
|------|------|
| 予測アルゴリズム | 直近1時間の使用率増加率の線形外挿 |
| 表示場所 | ポップオーバー（ゲージ下部） |
| 精度表示 | 「予測」ラベルを明示し、参考値であることを伝える |

**考慮事項**: 使用パターンは非線形であることが多い（バースト的な使い方）。単純な線形外挿では精度が低い場面がある点をUIで伝える。

### Shortcuts.app連携

**目的**: ユーザーが独自の自動化ワークフローを構築可能にする。例: 「使用率85%超で集中モードをON」「朝の使用率をSlackに投稿」。

**実装アプローチ**:

App Intentsフレームワークを使用して、macOS Shortcuts.appから使用率情報を取得可能にする。

| アクション | 説明 |
|-----------|------|
| 「Claude Code使用率を取得」 | 現在の5時間枠/7日枠の使用率を返す |
| 「使用率が高い場合に通知」 | 指定閾値を超えた場合にカスタムアクションを実行 |

**考慮事項**: App Intentsの登録にはmacOS 13+が必要。Siri連携（「Claude Codeの使用率は？」と声で確認）も自動的に有効になる。

### ヘルスチェック通知（長時間データ未更新時）

**目的**: collectorが正常に動作していない状態を早期に検出し、ユーザーに修復を促す。

**実装アプローチ**:

| 経過時間 | 通知内容 |
|---------|---------|
| 1時間 | 「Claude Codeからのデータ更新が1時間途絶えています」 |
| 24時間 | 「collectorが正常に動作していない可能性があります。設定を確認してください」 |

**考慮事項**: Claude Codeを使用していない時間帯（就寝中等）に不要な通知が飛ばないよう、「アクティブ時間帯」設定またはmacOSの集中モード連携を検討する。
